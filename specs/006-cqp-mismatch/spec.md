# Feature Specification: CQP Value Coherence Across Editor, Log, and Shared-Encoder Consumer

**Feature Branch**: `006-cqp-mismatch`

**Created**: 2026-05-24

**Status**: Draft

**Input**: User description: "CQP value mismatch between editor, log, and shared-encoder consumer"

## Clarifications

### Session 2026-05-24

- Q: Editor display ↔ encoder/log coherence — which surfaces must agree? → A: Four-way coherence (editor display = persisted config = encoder = slot start log); any divergence among the four surfaces is a defect.
- Q: Migration policy for existing on-disk consumer slots with stale standalone rate-control fields? → A: Normalize on load (clear or replace with an explicit "inherited" sentinel before any other code reads them; next save persists the normalized form).
- Q: For a consumer slot's editor, should the owner's current effective rate-control mode and value be surfaced inside that editor? → A: Yes — show them read-only, labeled "inherited from <owner name>"; fall back values must be shown when the owner's encoder construction fell back.
- Q: Which slot start modes does the coherence contract apply to (record-only, replay-only, rec+replay)? → A: All three. The contract applies to every slot start, regardless of which output is wired to the encoder. Spec wording standardizes on "slot start log" (formerly worded "start-of-recording log").
- Q: What should happen when a persisted rate-control value is outside the current encoder's reported range for the current rate-control mode (the most likely root cause of the owner-side editor=18 vs log=20 mismatch — the editor silently clamps the display while cfg keeps the original)? → A: Clamp on load, write back, warn — at load time, clamp the value to the nearest valid value for the current mode/encoder, emit a single warning log line identifying the slot, the original value, the encoder, the range, and the clamped value, and persist the clamped value at the next save. The editor sees the clamped value too; the "single source of truth" contract holds.
- Q: Scope of "consumer's own rate-control fields must not affect the slot" — does it cover just the log line, or every consumer-side read (investigation found the replay-buffer memory-cap estimate at `src/slot.cpp:748` also reads the consumer's stale fields)? → A: Uniform redirect — every consumer-side read of `rate_control` / `rc_value` (logging, replay-buffer sizing, and any future operational or user-visible surface) MUST resolve to the owner's effective values. The consumer's own fields are never read for any decision.
- Q: What should happen when the persisted `rate_control` mode string is not in the current encoder's reported list (a third silent-clobber path: the editor silently picks index 0 of the combo, and the next on_accept overwrites the saved mode)? → A: Same load-time normalize-and-warn pattern as FR-013, with the substitution rule being "first listed mode of the current encoder." Re-validate the value against the new mode's range per FR-013. Editor sees the substituted mode; cfg gets the substituted mode; next save persists it.
- Q: When the user changes a consumer slot's encoder combo back to a real (non-shared) encoder inside the editor — after the slot was previously saved as a consumer (so its persisted `rate_control` is the `"<inherited>"` sentinel and `rc_value` is `0`) — what should the editor present in the now-re-enabled rate-control rows? → A: The editor MUST seed the rate-control mode to the newly-selected encoder's first listed mode and the rate-control value to the midpoint of that mode's introspected range (clamped to the range; for lossless modes the value is `0` and the spinbox shows the lossless special text). The sentinel `"<inherited>"` MUST NEVER be presented as a selectable mode in any encoder's combo. The seeded values become the standalone values; the user can edit them before clicking Save.
- Q: When the editor renders the inherited rows for a consumer slot but the owner's encoder has not been constructed yet (owner stopped, no consumer has started since load, no `SharedEncoder` row exists), there are no "currently in effect" values to show. What should the editor display? → A: Show the owner's persisted (or load-time-normalized) `rate_control` and `rc_value` with the label `"(inherited from <owner>)"` and NO `[CBR fallback]` suffix. The `[CBR fallback]` suffix is reserved for the case where a `SharedEncoder` instance currently exists for the owner and reports `encoder_fallback_ == true`. This preserves FR-011 by reporting the same values the encoder WILL receive if built right now; if a later build falls back, the editor refresh (on next `update_shared_encoder_visibility` fire) will pick up the fallback flag and add the suffix.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Consumer slot reports the encoder it actually uses (Priority: P1)

A user has two slots: an "owner" slot that owns a video encoder configured for quality-based rate control (for example, CQP at quality 23), and a "consumer" slot that is configured to reuse the owner's encoder rather than create its own. The consumer slot was originally created with bitrate-based rate control (for example, CBR at 6000 kbps) before being switched to reuse the owner's encoder, so its stored rate-control mode and value are now stale and unused.

When the user starts the consumer slot, they should see in the application log the rate-control mode and value that the encoder is actually running with — i.e., the owner's settings — not the stale values that the consumer slot is still carrying from its earlier configuration.

**Why this priority**: This is the primary user-visible symptom of the bug. The slot start log line is the user's main source of truth for "what is this recording actually using"; if it lies, every downstream conclusion the user draws (about quality, file size, troubleshooting) is wrong. Fixing this restores trust in the log.

**Independent Test**: Configure slot A as the owner with CQP/23. Configure slot B with CBR/6000, then switch slot B to "Use slot A's encoder" and save. Start slot B and inspect the log line emitted at start. The reported rate-control mode and value must match what the shared encoder is actually using (CQP/23), not the consumer's stale stored values (CBR/6000).

**Acceptance Scenarios**:

1. **Given** consumer slot B has stale rate-control fields from a prior standalone configuration **and** B reuses owner slot A's encoder, **When** B is started, **Then** the slot start log line for B reports the same rate-control mode and value that A's encoder was built with.
2. **Given** owner slot A is currently running with CQP/23 and consumer B reuses A's encoder, **When** B is started, **Then** B's start log line reports CQP/23, not B's stored values.
3. **Given** the owner's encoder fell back to a safer rate-control mode at construction (e.g., the requested encoder was unavailable and the build path forced CBR with a safe bitrate), **When** any slot in that group is started, **Then** the start log for that slot reports the fallback rate-control mode and value actually in effect — not the originally requested values.

### User Story 2 - Editor does not present stale rate-control values for consumer slots (Priority: P1)

A user opens the editor for a consumer slot (a slot whose video encoder is "Use slot X's encoder"). The editor currently hides the rate-control and rate-control-value rows because they would not be applied if the user edited them, but the underlying stored values for that slot are stale and continue to surface in other places (the log, persisted save data). The user must not be misled into believing those stored values affect this slot.

When viewing or editing a consumer slot, the editor must make it unambiguous that rate-control settings come from the owner, and saving the slot must not leave it in a state where stale values can later be reported as if they were in effect.

**Why this priority**: Tied for P1 with Story 1 because editor and log are the two surfaces where the user encounters the inconsistency. If the editor leaves stale values in the saved config, even a fix to the log will re-break the next time anyone reads the config directly (e.g., diagnostics, support bundles, future features).

**Independent Test**: Create slot B with CBR/6000, switch it to reuse slot A's encoder, save it, close the editor, reopen the slot's editor, and inspect the persisted configuration (the same data structure that the log and any future readers consume). The persisted rate-control mode and value for B must be cleared or replaced with the "inherited" sentinel; they must never silently retain B's pre-switch CBR/6000, and they must not carry a snapshot copy of slot A's values that could drift from A's later edits.

**Acceptance Scenarios**:

1. **Given** a user changes slot B from standalone (CBR/6000) to reusing slot A's encoder and clicks Save, **When** B's stored config is re-read, **Then** B's rate-control mode and value are cleared or set to the "inherited" sentinel — never a stale standalone value the user did not just authorize, and never a snapshot copy of slot A's values that could drift.
2. **Given** a user opens the editor for an existing consumer slot, **When** the editor renders, **Then** the user sees, for the rate-control mode and value, the owner's current effective values shown read-only and labeled with the owner slot's name (per FR-005), so the absence of editable fields is not mistaken for "the slot has no rate-control settings."
3. **Given** a saved configuration on disk in which a consumer slot still has stale standalone rate-control fields from before this fix shipped, **When** the application loads that save, **Then** the in-memory configuration for that slot is normalized at load time (per FR-012), and the very first read by any surface (log, editor, diagnostics) sees the normalized form — never the on-disk stale values.

### User Story 3 - Owner's quality-mode value reaches the encoder under the same key the editor wrote (Priority: P2)

A user configures the owner slot with a quality-based mode (CQP, CRF, ICQ, etc.) and a specific quality value. The editor reads the legal range for that value by introspecting the encoder's properties (so the range and labels match the encoder), but at encoder-build time the value is written under whichever single quality key the encoder exposes (`crf`, `cqp`, `cq_level`, `qp`, `icq_quality`, `global_quality`) or split across `qpi`/`qpp`/`qpb` for QSV. If the editor introspected a range under one key while the build path writes under a different key (or vice versa), the user's typed value can either be silently clamped to an inappropriate range in the UI or written into a key the encoder does not honor.

The editor's quality range and the build path's quality value must be derived from the same key (or set of keys) on the same encoder, so the value the user types in the editor is the value the encoder receives.

**Why this priority**: P2 because today's code path appears to keep the editor and build paths aligned by both probing the same key list, but the alignment is implicit and easy to drift; making it an explicit, tested invariant prevents future regressions and the same "values disagree" class of bug recurring on the owner side.

**Independent Test**: For each supported video encoder ID (x264, NVENC variants, AMF variants, QSV variants, VideoToolbox, AV1 variants), the editor's introspected quality range and the key written at encoder-build time must reference the same encoder property; the test must fail if any encoder introspects under key X for range purposes but writes the value under a different key Y.

**Acceptance Scenarios**:

1. **Given** any supported video encoder selected in the editor under a quality-based rate-control mode, **When** the user enters a value and saves, **Then** the encoder receives that exact value under the same property key (or key set) whose range the editor used to bound the input.
2. **Given** an encoder that splits its quality across multiple keys (e.g., QSV `qpi`/`qpp`/`qpb`), **When** the user enters a single quality value, **Then** all of the encoder's quality-related keys are set as a unit and the editor's range reflects the same set.

### Edge Cases

- **Owner's encoder unavailable on the host machine**: When the configured shared encoder cannot be created and the build path falls back to a different encoder and rate-control mode (e.g., obs_x264 + CBR with a safe bitrate), the start log for both the owner and any consumer must report the fallback values that are actually running — not the originally requested ones.
- **Owner not running, consumer is**: A consumer can start while its owner slot is stopped, because the shared encoder is built from the owner's persisted config. The reported rate control must come from that persisted owner config (or its fallback), not from the consumer's stale fields.
- **User switches a slot from standalone to consumer, then back to standalone**: The slot's rate-control mode and value must be recoverable to sensible values on the return trip — either preserved from before the switch, defaulted, or surfaced as something the editor prompts the user to confirm. They must not become silently invalid. Per FR-016, when the editor re-enables the rate-control rows after the user re-selects a non-shared encoder, the rows are seeded to the new encoder's first listed mode and the midpoint of that mode's range; the user can adjust before saving.
- **Owner reconfigures CQP from 23 to 18 while consumers exist**: After the owner is saved with the new value, the next start of any slot in that group must report the new value. No slot should log the pre-change value as if it were in effect.
- **Lossless mode on the owner**: When the owner's effective mode is Lossless (no numeric value), the log for owner and consumers must indicate "Lossless" (or equivalent) rather than printing a numeric value that has no meaning.
- **Stored mode string is bitrate-based but stored numeric value is in the quality range (or vice versa)**: Legacy saves or hand-edited configs may pair a quality-mode string with a kbps-shaped number, or a bitrate-mode string with a CQP-shaped number. The log must not present the number as if it matched the mode when it does not.
- **Consumer slot's replay buffer memory cap**: When a consumer slot has a replay buffer enabled, the buffer's memory-cap estimate (a function of `is_bitrate_based(mode) ? rc_value : <fallback>`) must be computed from the owner's effective rate-control mode and value, not from the consumer's own stale fields. A consumer whose stale fields say one thing while the owner runs another must not allocate a buffer sized to the stale number.
- **Owner slot deleted while a consumer survives (orphan consumer)**: A consumer slot may persist on disk whose `shared_encoder_slot_id` no longer resolves because the user deleted the owner without reverting the consumer first. On the next read by any surface (slot start, editor open, replay-buffer estimate), the resolution helper MUST return safe last-resort values — rate-control mode `CBR`, value `6000` (the same defaults used by load-time back-compat), `fallback == false`, and an empty owner name — without crashing. The slot start log MUST emit those values; the editor, when opened on such an orphan consumer, MUST show those last-resort values in the inherited rows with a label clarifying the dangling reference (e.g., `(inherited — owner missing)`). A single warning log line per orphan SHOULD be emitted on load so the user is alerted to the dangling reference; the orphan state is otherwise non-fatal.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST emit, at the moment a slot starts (regardless of whether the slot is record-only, replay-only, or rec+replay), a log line whose rate-control mode and rate-control value reflect the values actually configured on the encoder the slot is using.
- **FR-002**: For a consumer slot (one that reuses another slot's encoder), the slot start log MUST report the owner slot's effective rate-control mode and value, not the consumer's own stored fields.
- **FR-003**: If the encoder construction path falls back to a different encoder or rate-control mode than the user configured (e.g., requested encoder unavailable), the slot start log MUST report the fallback values that are running, not the originally requested ones.
- **FR-004**: A consumer slot's persisted rate-control mode and rate-control value MUST NOT be reported anywhere in the application's user-visible surfaces (logs, editor labels, diagnostics) as if they were in effect for that slot.
- **FR-005**: The editor MUST, for a consumer slot, display the owner's effective rate-control mode and value as read-only fields labeled to identify the owner slot by name (e.g., "inherited from Slot 1"). "Effective" resolves as follows: if a `SharedEncoder` instance currently exists for the owner, the displayed values are the encoder's running values, and if `encoder_fallback_` is set, the label additionally carries a `[CBR fallback]` suffix; if no `SharedEncoder` instance exists yet (owner stopped, no consumer started since load), the displayed values are the owner's persisted (load-time-normalized) `rate_control` / `rc_value` without any fallback suffix.
- **FR-006**: When the user saves a consumer slot, the system MUST leave the slot in a state where future readers of the slot's stored configuration cannot mistake the slot's residual standalone rate-control values for values that are in effect.
- **FR-007**: For an owner slot in a quality-based rate-control mode, the numeric range presented in the editor and the property key under which the value is written at encoder-build time MUST be derived from the same encoder property (or coordinated set of properties), per encoder, so that the value the user enters is the value the encoder receives.
- **FR-008**: When the owner's effective rate-control mode takes no numeric value (Lossless), the slot start log MUST indicate that fact rather than printing a numeric value.
- **FR-009**: When a user reconfigures the owner's rate-control mode or value and restarts any slot in the shared-encoder group, the next slot start log for that slot MUST reflect the updated values.
- **FR-010**: The system MUST behave consistently whether the owner slot is currently running or stopped at the time a consumer starts: the values reported by the consumer's start log MUST come from the owner's persisted (or fallback) configuration in both cases.
- **FR-011**: For any slot at any moment, the rate-control mode and value displayed in the editor, the values persisted in the slot's configuration, the values applied to the encoder at start, and the values reported in the slot start log MUST all be the same single value. Divergence among any of these four surfaces is a defect, including (but not limited to) the owner-side editor=18 vs log=20 case in the original bug report.
- **FR-012**: When the application loads a saved configuration, the system MUST normalize any consumer slot whose persisted standalone rate-control fields are stale, so that subsequent reads of the in-memory configuration (by the log, the editor, diagnostics, or any other surface) cannot return those stale values. The next time the user saves the configuration, the normalized form MUST be the form written to disk.
- **FR-013**: When the application loads a saved configuration and a slot's persisted rate-control value is outside the range the current encoder reports for the slot's current rate-control mode, the system MUST clamp the value to the nearest valid value, emit a single warning log line identifying the slot name, the original persisted value, the encoder, the reported range, and the clamped value, and persist the clamped value the next time the configuration is saved. The clamped value is the value the editor displays, the value the encoder receives, and the value reported in the slot start log — preserving the four-way coherence contract (FR-011). This applies to quality-based modes (using the encoder's introspected quality range) and to bitrate-based modes (using the encoder's introspected bitrate range) alike. For consumer slots (where `shared_encoder_slot_id` is non-empty), the consumer-clear normalization in FR-012 runs first and replaces the persisted value with the sentinel and `0`; the clamp does not apply to the consumer's own persisted value (the owner's value is what reaches the encoder, and the owner's own load-time normalization clamps it under this same rule).
- **FR-014**: For a consumer slot, no code path MAY read the consumer's own `rate_control` or `rc_value` fields for any operational decision (including replay-buffer memory-cap estimation, bitrate estimation, output configuration), user-visible surface (logs, editor, diagnostics), or downstream side effect. Every such read MUST resolve to the owner slot's effective rate-control mode and value (or the fallback values, if the owner's encoder construction fell back). This generalizes FR-002 from the slot start log to every read site.
- **FR-015**: When the application loads a saved configuration and a slot's persisted `rate_control` mode string is not in the rate-control list reported by the slot's current encoder, the system MUST substitute the encoder's first listed mode, emit a single warning log line identifying the slot name, the original persisted mode, the encoder, and the substituted mode, re-validate the rate-control value against the substituted mode's range (applying FR-013 if the value is out of range under the new mode), and persist the substituted mode at the next save. The substituted mode is the mode the editor displays, the mode the encoder receives, and the mode reported in the slot start log — preserving FR-011.
- **FR-016**: When the user changes a slot's video-encoder selection in the editor from a shared-encoder choice (`"shared:<owner-slot-id>"`) back to a real encoder — typically a consumer slot whose persisted `rate_control` is the `"<inherited>"` sentinel with `rc_value == 0`, but the rule applies regardless of the prior persisted shape — the editor MUST repopulate the slot's rate-control mode and value with valid defaults for the newly-selected encoder before the user clicks Save. The mode defaults to the first item the encoder reports for its `rate_control` property; the value defaults to the midpoint of that mode's introspected range, clamped to that range (or `0` for lossless modes, with the spinbox special-value text shown). When the encoder reports no range for the seeded mode (introspection returns no min/max), the mode MUST still be seeded and the spinbox MUST be left at its current minimum value — the user can correct the value before clicking Save. The sentinel `"<inherited>"` MUST NEVER appear as a selectable rate-control mode in any editor combo. The seeded mode and value become the standalone values written when the user clicks Save; the user MAY edit them before saving. This closes the consumer → standalone transition gap left by FR-006 (save-side sentinel write) and preserves FR-011 four-way coherence on the return trip: editor display, persisted config (on next Save), encoder, and slot start log all share one value.

### Key Entities *(include if feature involves data)*

- **Slot configuration**: The persisted set of fields for one recording slot. Relevant fields for this feature: the slot's identity, an optional reference to another slot whose encoder it reuses, the rate-control mode string (e.g., CBR, CQP, CRF, Lossless), and the rate-control numeric value (kbps for bitrate modes, quality level for quality modes).
- **Shared encoder group**: The set of slots that all use the same underlying video encoder. Exactly one slot in the group is the owner (the slot whose configuration was used to construct the encoder); the rest are consumers whose own rate-control fields do not affect the encoder.
- **Effective rate-control state**: For any slot at any moment, the rate-control mode and value that the encoder it uses is actually running with — which may differ from the slot's stored fields (for consumers) and from the user's originally requested values (when the build path fell back due to unavailable encoder).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For 100% of slot start events across all configurations (standalone, owner, consumer, fallback), the rate-control mode and value displayed in the editor, persisted in the slot's configuration, applied to the encoder, and reported in the slot start log are the same single value.
- **SC-002**: A user opening the editor for a consumer slot can identify, within 5 seconds and without external documentation, (a) that rate-control settings are inherited, (b) which owner slot they are inherited from by name, and (c) the actual rate-control mode and value currently in effect (including any fallback).
- **SC-003**: Zero slot configurations on disk, after saving a consumer slot through the editor, contain residual standalone rate-control fields that would be misreported as "in effect" by any user-visible surface.
- **SC-004**: For every supported video encoder, the quality value the user enters in the editor reaches the encoder unchanged — no silent clamping caused by range/key mismatch, no value written under a key the encoder ignores. The guarantee is **structural**: the editor's range introspection and the encoder-build write path walk the same shared quality-key list, so any encoder reachable through that list is covered by construction. **Manual verification scope**: Windows-only (the maintainer's test environment), covering the encoders available there (typically NVENC + `obs_x264`; AMF / QSV where the hardware exists). macOS VideoToolbox and Ubuntu `obs_x264 + CRF` are NOT manually verified by the maintainer; cross-platform correctness rests on the structural guarantee and on future community reports.
- **SC-005**: After the owner's rate-control mode or value is changed and saved, the next start of any slot in that group reports the new values in its log line — verified within one start cycle, with no stale value reported.

## Assumptions

- Existing user behavior of consulting the slot start log line as the authoritative "what is this recording using" indicator is preserved; the fix changes what that line says, not whether or where it is emitted. The line is emitted for record-only, replay-only, and rec+replay slot starts alike, and the coherence contract applies uniformly to all three.
- The existing UI pattern of hiding rate-control rows entirely for consumer slots is acceptable to keep, provided the editor adds an explicit indicator naming the owner (FR-005); a full redesign of the consumer-slot editor is out of scope.
- "Editor" in this spec refers to the per-slot editor dialog where a user configures a single slot, not the broader OBS Studio settings UI.
- The set of supported video encoders is the set the project already exposes today (x264, NVENC, AMF, QSV, VideoToolbox/Apple, plus AV1 variants of those); no new encoders are introduced by this work.
- Persisted-config migration of existing user saves happens at load time: when a save is read, any consumer slot's stale standalone rate-control fields are normalized (cleared or replaced with an explicit "inherited" sentinel) before any other code can read them, and the next save persists the normalized form. Users are not required to re-open each affected slot in the editor for the migration to take effect.
- The application's logging level and format conventions (the existing `[multi-scene-rec]` prefix, INFO level for start, etc.) are kept; only the content of the rate-control portion of the start line changes.
