# Feature Specification: Replay file uniqueness across slots sharing an output directory

**Feature Branch**: `007-fix-replay-collision`

**Created**: 2026-05-25

**Status**: Draft

**Input**: User description: "Replay file silently lost when multiple slots share an output directory"

## Clarifications

### Session 2026-05-25

- Q: Include the misleading "replay save OK" log-line fix in this feature, or defer to a separate one? → A: Include in this feature. The scope is one coherent "no silent replay loss" guarantee covering both the filename-collision cause (US1) and the deceptive plugin log line that hides the loss when it does occur — from the cross-slot collision case AND from other write-failure causes (disk full, permissions, FS race, OBS-internal muxer errors that don't surface through the proc return value).
- Q: Which identifier goes in the replay filename — sanitized user-facing slot name (readable), slot persisted id (collision-proof), name + id always, or name-only-with-id-on-conflict? → A: Name + short id always. Every replay filename embeds the sanitized user-facing slot name AND a short stable prefix of the slot's persisted opaque id, in that order, on every save. Collision across slots is impossible by construction — no runtime conflict detection. Readable attribution at a glance is preserved because the user-facing name is the first component. Example shape: `CamA_18f3a2_Replay_2026-05-23_22-54-03.mkv`.
- Q: Sanitization policy when the slot's user-facing name contains path-illegal characters (or matches a Windows reserved name) — silent strip, strip-and-warn once per session, strip-and-warn every save, or strip-and-persist? → A: Silent strip. The slot's stored user-facing name is untouched; the version embedded in the constructed replay filename has each non-portable character replaced with a fixed safe placeholder. No warning log is emitted on every save (the truthful path-in-log line from FR-012 already shows the resulting file path, which is the user's verification channel). The recording filename construction at `src/slot.cpp:96-103` is unchanged per FR-006/FR-008 — the asymmetry (recording does not sanitize, replay does) is accepted and noted in Assumptions.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Two slots saving replays into the same directory keep both files (Priority: P1)

A user has two recording slots, each with the replay buffer enabled. Both slots are intentionally configured with the same output directory — for example, the user wants all "match" replays from two camera angles to land in one folder for easy review. The user (or two automation triggers) press Save Replay on each slot within the same calendar second.

Today, both slots' replay outputs write to the same target file name in that directory because the replay file name carries only a timestamp at second resolution and no slot identifier. The second writer overwrites the first, and one of the user's clips is silently and permanently lost — there is no error log, no UI indication, and no way to recover the missing clip after the fact.

After this fix, both replay files MUST be present on disk after the two saves complete, distinguishable by name, regardless of whether the saves landed in the same second.

**Why this priority**: P1 because the failure mode is data loss without any warning. The user has no way to know a clip was lost short of counting the files they expected versus the files that exist, and the lost clip cannot be recovered. Every other concern (file naming aesthetics, configuration discoverability) is secondary to "the user's recorded data must not disappear."

**Independent Test**: Configure two slots — call them "CamA" and "CamB" — each with the replay buffer enabled, both with the same output directory and the same container format. Start both slots, wait long enough for their replay buffers to fill, then trigger Save Replay on both within the same calendar second (e.g., via two hotkeys pressed in rapid succession). Inspect the output directory. Two distinct replay files MUST be present, one corresponding to each slot. Repeat with three or more slots; the count of files MUST equal the count of saves.

**Acceptance Scenarios**:

1. **Given** two slots with the replay buffer enabled, both pointed at the same output directory, both using the same container format, **When** Save Replay is triggered on each slot within the same calendar second, **Then** two distinct replay files are present in the directory after both saves complete, and neither save overwrites the other.
2. **Given** three or more slots configured as in scenario 1, **When** Save Replay is triggered on all of them within the same calendar second, **Then** the count of newly-created replay files equals the count of triggered saves, with no silent overwrites.
3. **Given** two slots pointed at the same output directory but using different container formats (e.g., mp4 vs mkv), **When** Save Replay is triggered on each within the same calendar second, **Then** both files are present (the existing container-suffix distinction is preserved, and the fix does not weaken it).

### User Story 2 - User can identify the producing slot from the replay file name alone (Priority: P2)

A user later opens the shared output directory and sees a list of replay files. They need to know which slot produced each file — for example, to attribute a clip to the right camera, or to delete only the clips from one slot — without opening the files or consulting the application.

Today, every replay file name across every slot follows the same `Replay_<timestamp>` pattern; nothing in the file name identifies the producing slot. The user has no way to attribute files except by opening them.

After this fix, the replay file name MUST embed an identifier that lets the user attribute the file to its producing slot at a glance.

**Why this priority**: P2 because correct attribution is the natural follow-on to "no silent loss." Once the user knows all their files are present, the next question is "which is which." The continuous recording output already includes the slot's user-facing name in its file name (`<slot-name>_<timestamp>.<ext>`); the replay output should follow the same convention for visual parity between a slot's recordings and its replays in the same directory.

**Independent Test**: With two slots named "CamA" and "CamB" sharing an output directory, trigger one Save Replay on each. Inspect the directory. Reading only the file names (no other tool, no metadata, no opening the files), the user MUST be able to state which file came from CamA and which came from CamB.

**Acceptance Scenarios**:

1. **Given** two slots named with distinct user-facing names, each pointed at the same output directory, **When** the user inspects a saved replay file's name, **Then** the user-facing slot name appears as a component of the file name in a position that is consistent across all replay saves and matches the position used by the continuous recording file name (for visual grouping when both file types co-exist).
2. **Given** the user-facing slot name is empty, **When** a replay is saved, **Then** the file name still uniquely identifies the slot using a stable fallback identifier (rather than producing an ambiguous file name like `Replay_<timestamp>` that could collide with another slot's empty-named save).
3. **Given** two slots that happen to share the same user-facing name in the same configuration AND share the same output directory, **When** Save Replay is triggered on each within the same calendar second, **Then** the resulting file names are still distinct because the id-derived component of the file name (FR-003 (b)) differs between the two slots and is always present — uniqueness does NOT depend on runtime conflict detection or on the user-facing name being unique across the configuration.

### User Story 3 - Replay-save log line truthfully reflects whether a file was written (Priority: P2)

A user triggers Save Replay on a slot via hotkey or button. Today, `src/slot.cpp:1066` emits `'<slot>' replay save OK` whenever the OBS replay-buffer `save` proc handler call returns `true` — which only confirms the proc was dispatched, NOT that any file was written. If the underlying write fails after dispatch (cross-slot filename collision, Windows share-violation race, disk full, output path missing or read-only, OBS-internal muxer error not surfaced through the proc return value), the plugin still logs OK and the user has no signal that a clip was lost.

After this fix, the plugin's replay-save log line MUST truthfully reflect whether the save produced a file. Wording such as "OK" or "succeeded" MUST NOT appear unless the plugin has confirmed a file was written. When confirmation is unavailable at the time of logging (the proc has dispatched but the outcome is not yet known), the line MUST use neutral wording — e.g., "save requested" — that does not imply success. When confirmation IS available, the line SHOULD identify the on-disk path of the written file so the user can verify it.

**Why this priority**: P2 — fixing US1 eliminates the most common silent-loss cause (cross-slot filename collision), but does not by itself cover other causes of silent loss (Windows FS races, disk full, permission denial, missing directories, OBS-internal write failures). Without US3, US1 ships with a plugin log line that will continue to lie any time something else goes wrong, and the user will continue to have no observable signal of data loss. US3 closes the observability gap so any future silent-loss event is detectable from the plugin's own log.

**Independent Test**: Trigger a Save Replay under conditions that cause OBS to fail the actual file write while the `save` proc still returns true — for example, configure the slot's output directory to a path that exists at slot start but is deleted (or revoked permissions) just before the save is triggered. Inspect the plugin log. The log line for that save MUST NOT use success wording. Under normal successful saves, the log line MUST identify the file path in a form the user can copy-paste to locate the file.

**Acceptance Scenarios**:

1. **Given** the OBS replay-buffer `save` proc handler returns true but no file is ultimately written to disk (any cause), **When** the plugin emits its log line for that save, **Then** the line does NOT use wording that implies success — it either reports the actual failure (when the failure is detectable in time) or uses neutral wording such as "save requested" (when the outcome is not yet known at log time).
2. **Given** a Save Replay completes successfully and produces a file on disk, **When** the plugin emits its log line for that save, **Then** the line identifies the on-disk path of the file (the same path the OBS muxer's own `[ffmpeg muxer: 'replay_out_<id>'] Wrote replay buffer to '<path>'` line names), in a form the user can use to locate the file.
3. **Given** a Save Replay is triggered against a slot whose configured output directory does not exist or is not writable, **When** the plugin emits its log line for that save, **Then** the line reports the failure (not success wording), naming at minimum the slot.

### Edge Cases

- **Same-second simultaneous saves across N slots sharing a directory**: All N files MUST be present after the saves complete. The guarantee is not "two slots are safe"; it is "any number of slots is safe."
- **Two slots share both the user-facing name and the output directory**: Slot names are not enforced unique in this plugin. The replay file name MUST still distinguish the two slots' saves — and per FR-003 / FR-004 this distinction is structural (the id-derived component is unconditional and always differs across slots), not runtime-detected.
- **Empty user-facing slot name**: The replay file name MUST still uniquely identify the slot. Per FR-003, when the user-facing name component is empty the id-derived component alone carries the slot identity (optionally with a fixed literal placeholder consistent with the existing recording-filename fallback at `src/slot.cpp:96-104`).
- **Slot name contains only characters removed by FR-004a sanitization**: The slot-identifying component degenerates to component (b) — the id-derived prefix — alone, exactly as in the empty-name case. The filename remains unique by construction and the path-portability guarantee is preserved.
- **Slot rename after a replay has been saved**: The on-disk file keeps the name it was given at save time. The application MUST NOT retroactively rename, move, or delete previously saved replay files.
- **Existing on-disk replay files from before this fix shipped**: Pre-fix files use the `Replay_<timestamp>` pattern with no slot identity. They remain on disk under their original names. The application MUST NOT migrate, rename, or delete them. Mixed directories (pre-fix and post-fix files co-existing) are an expected state and MUST NOT cause errors at scan, load, or save time.
- **One slot saves two replays in rapid succession (intra-slot same-second)**: This is OBS's replay-buffer-internal timing behavior, not the cross-slot collision this fix targets. The fix MUST NOT regress intra-slot behavior (whatever OBS does today — typically the second save is delayed slightly by the replay buffer's own internal serialization — MUST continue to work).
- **Filesystem case-insensitivity (Windows, macOS default)**: Two slots whose user-facing names differ only by case (e.g., "CamA" and "camA") MUST still produce distinct replay file names. Per FR-003 / FR-004, the id-derived component is unconditional and always differs across slots, so this case is structurally safe regardless of how the underlying filesystem compares case.
- **Slot name contains filesystem-reserved characters (e.g., `<>:"/\|?*` on Windows, or a reserved name like `CON` / `PRN`)**: Per FR-004a, those characters are sanitized inside the replay filename only — replaced with a fixed safe placeholder. The slot's persisted user-facing name is untouched. The continuous recording filename construction (`src/slot.cpp:96-104`) is out of scope per FR-006/FR-008 and continues to embed the raw `cfg.name`; the asymmetry is intentional and noted in Assumptions.
- **Replay save triggered programmatically vs via hotkey/button**: The fix MUST apply regardless of how the save is triggered. The trigger path does not affect file name construction.
- **Slot is in replay-only mode (no continuous recording file)**: The same naming guarantees apply. The absence of a recording file in the directory does not change the replay file name pattern.
- **Save proc returns true but no file is written**: The OBS replay-buffer `save` proc handler dispatching successfully does NOT imply a file was written. This dissociation can occur from a cross-slot filename collision (the case FR-001/FR-002 fix at the source), a Windows share-violation race, disk-full, output-directory missing or read-only at save time, or any OBS-internal muxer failure that doesn't propagate through the proc return value. The plugin's log line for the save MUST NOT use success wording in any of these cases — see FR-011.
- **Output directory does not exist at save time**: Same as above — the plugin's `save` proc call may return true while OBS fails the actual write. The log line MUST reflect the failure, not the proc-handler-dispatched status.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: When a replay is saved, the resulting replay file name MUST embed a slot-identifying component that distinguishes the producing slot from every other slot in the current configuration, so that no two slots can produce identical replay file names in the same directory.
- **FR-002**: When two or more slots are configured with the same output directory and the same container format, and Save Replay is triggered on each within the same calendar second, all of the resulting replay files MUST be preserved on disk — no file may be silently overwritten by another slot's save.
- **FR-003**: The slot-identifying component of the replay file name MUST consist of, in order, (a) the slot's user-facing name when non-empty (after the sanitization in FR-004a) and (b) a short stable prefix derived from the slot's persisted opaque id. Both parts appear on every save; component (b) is NEVER omitted. When the user-facing name is empty, component (a) is skipped (or replaced with a fixed literal placeholder consistent with the existing recording-filename fallback at `src/slot.cpp:96-104`) and component (b) alone identifies the slot.
- **FR-004**: Uniqueness across slots in the same configuration MUST be a structural guarantee of the filename construction itself, not a property that depends on runtime conflict detection or on the user-facing slot names being unique across the configuration. Two slots that happen to share a user-facing name MUST still produce distinct replay file names by construction, because the id-derived component (FR-003 (b)) differs between them. The implementation MUST NOT include conditional "if names clash, append id" logic — the id-derived component is unconditional.
- **FR-004a**: Before the user-facing slot name is embedded in the replay file name (FR-003 (a)), it MUST be sanitized to remove characters that are not portable across the filesystems of the supported platforms (Windows, macOS, Linux). Sanitization replaces each non-portable character with a fixed safe placeholder character (or removes it; implementation choice) and MAY also collapse repeated placeholder characters. Sanitization is silent in the routine case — no warning log line is emitted on every save — to avoid log noise; the truthful log line from FR-012 already surfaces the resulting path so the user can verify what was written. The slot's stored user-facing name on disk is NOT modified by this sanitization; only the version that appears inside the constructed file name is transformed.
- **FR-005**: The slot-identifying component MUST appear in the file name in the same position relative to the timestamp that the continuous recording file name uses (slot identity first, then timestamp), so that a slot's recordings and replays sort and group together when both file types co-exist in the same directory.
- **FR-006**: The fix MUST NOT alter the file name pattern of the continuous (non-replay) recording output. The recording filename construction site is out of scope for this change.
- **FR-007**: The fix MUST NOT alter, migrate, rename, move, or delete any pre-existing replay file on disk that was created before this fix shipped. Pre-fix and post-fix files MUST be permitted to co-exist in the same directory without producing errors at scan, load, or save time.
- **FR-008**: The fix MUST NOT alter any user-configurable persisted slot setting — output directory, container, replay buffer length, replay buffer size, or any other field of the slot configuration. The change is confined to the runtime construction of the replay file name passed to the replay-buffer output.
- **FR-009**: Renaming a slot MUST NOT retroactively rename or remove previously saved replay files. Only replay files saved after the rename use the new name; earlier files keep the name they were given at save time.
- **FR-010**: The uniqueness and identification guarantees in FR-001 through FR-004 MUST apply regardless of how the save is triggered (manual hotkey, manual button, programmatic / future-API call) and regardless of whether the slot is in record-only-plus-replay mode or replay-only mode.
- **FR-011**: When the plugin emits a log line for a Save Replay operation, the wording MUST truthfully reflect what the plugin has actually confirmed. Wording that implies success ("OK", "succeeded", "saved", or any synonymous claim) MUST NOT appear unless the plugin has confirmed a file was written. When the plugin has dispatched the save but the outcome is not yet known (or cannot be known given the available OBS API surface), the line MUST use neutral wording — for example, "save requested" — that does not imply success. The current `'<slot>' replay save OK` wording, which is emitted whenever the `save` proc handler call returns true without confirming a write, MUST be replaced.
- **FR-012**: When the plugin has confirmed that a Save Replay produced a file on disk, the log line for that save SHOULD identify the on-disk path of the file in a form the user can use to locate it — at minimum the same path the OBS muxer's own `[ffmpeg muxer: 'replay_out_<id>'] Wrote replay buffer to '<path>'` line names for the same save. This makes the plugin's log self-sufficient for verification and removes the requirement that the user cross-reference the OBS log to confirm where a file landed.
- **FR-013**: The plugin's log line for a Save Replay MUST NOT contradict the OBS muxer's own write log for the same save. If OBS logs that a write succeeded and the plugin logs success for that save, the two log lines MUST agree on the path. If OBS does not log a write for the save, the plugin MUST NOT log success for it.

### Key Entities *(include if feature involves data)*

- **Slot configuration**: The persisted set of fields for one recording slot. Relevant fields for this feature: the slot's stable persisted identifier (opaque, generated once at slot creation, unique across slots in the same configuration), the slot's user-facing name (editable, not enforced unique), the slot's output directory, the slot's container format, and the per-slot replay buffer settings (enabled flag, buffered length).
- **Replay file**: A clip file produced by the replay buffer when the user (or an automated trigger) saves the current replay window. The file is written to the slot's output directory at the moment of the save. Its file name today follows a fixed `Replay_<timestamp>` pattern with no slot identity; this spec changes that pattern.
- **Output directory**: A user-configurable, per-slot filesystem path where the slot's recording and replay files are written. Multiple slots in the same configuration MAY point at the same directory; this is a supported configuration, not an error condition.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For every configuration in which two or more slots share an output directory and the same container format, simultaneous Save Replay events across those slots result in zero silent file overwrites between slots — every triggered save produces exactly one preserved file on disk.
- **SC-002**: A user inspecting a directory of saved replay files can correctly attribute each file to its producing slot for 100% of the files, using only the file name (no application, no metadata, no opening the file).
- **SC-003**: Across N slots configured with the same output directory and the same container format, the number of distinct replay file names producible by those slots is unbounded (i.e., the file-name space scales with the number of slots and the number of saves; there is no fixed-size bucket in which collisions become inevitable).
- **SC-004**: The continuous recording file name construction is unchanged by this work, verified by inspection: the recording path that was emitting `<slot-name>_<timestamp>.<ext>` before continues to emit the same shape after, with the same slot-name component, the same timestamp resolution, and the same container suffix.
- **SC-005**: Pre-fix replay files left on disk by earlier builds remain present and unmodified after this fix ships, and the application produces no errors when a directory contains a mix of pre-fix and post-fix replay files.
- **SC-006**: Zero plugin log lines emit success wording for a Save Replay operation that did not produce a file on disk. If a save fails for any reason, the user can detect that fact from the plugin's own log alone — they do not need to cross-reference the OBS muxer log or inspect the output directory to discover a silent loss.
- **SC-007**: For 100% of Save Replay operations that successfully produce a file, the plugin's log line for the save identifies the on-disk path of the file in a form the user can copy-paste to locate it.

## Assumptions

- The bug is specific to the per-slot replay buffer output managed by this plugin. The continuous recording filename (built at `src/slot.cpp:96-104` today) already embeds `cfg.name` and is therefore not affected by the cross-slot directory-share case at the level addressed here; it is explicitly out of scope. "Replay" in this spec refers only to the per-slot replay buffer; it does not refer to OBS Studio's separate main replay buffer feature.
- The slot's stable persisted identifier (the `id` field on the slot configuration, generated once at slot creation and persisted across saves, opaque to the user, unique within a configuration) is available at the moment the replay-buffer output is configured and is suitable as a uniqueness tiebreaker when the user-facing name alone would not guarantee uniqueness.
- Slot user-facing names are not enforced unique across slots in the same configuration today, and this spec does not introduce such an enforcement. The replay file-name construction provides the uniqueness guarantee instead.
- No new user-facing UI warning is required when two slots are configured with the same output directory. The collision is removed at the filename layer, which makes a configuration-time warning unnecessary; users who deliberately point multiple slots at one directory continue to do so without prompts.
- No retroactive migration of pre-existing on-disk replay files is performed. Old files keep their original `Replay_<timestamp>` names. Users who want consistent naming across old and new files can rename old files manually; the application does not touch them.
- The fix is scoped to runtime construction of the replay-buffer output's filename pattern. No new persisted save-format keys are introduced. The existing per-slot output-directory, container, and replay-length settings are preserved unchanged in shape and meaning.
- Slot names that contain filesystem-illegal characters previously caused the continuous recording path to fail at write time (the raw `cfg.name` is concatenated into the recording filename at `src/slot.cpp:96-104` without sanitization, so the muxer fails to create the file and surfaces an error via `obs_output_get_last_error` at the existing recording call sites). This spec introduces sanitization ONLY for the replay-filename construction path (FR-004a); the recording path is left unchanged per FR-006/FR-008. Bringing recording sanitization to parity is a separate-feature concern.
- The fix is verified manually on Windows (the maintainer's test environment). macOS and Ubuntu builds receive the same code path; cross-platform correctness rests on the platform-agnostic nature of the filename construction (it is a string-build operation passed to libobs's replay-buffer output settings, with no platform-specific branches) and on future community reports.
- Whether the plugin can confirm a Save Replay's outcome at log time depends on the OBS replay-buffer output's signal / proc surface (a `saved` signal, a per-write proc result, or equivalent). The spec describes the user-visible outcome required (FR-011/FR-012/FR-013); the precise mechanism — subscribing to a `saved` signal vs reading `obs_output_get_last_error` vs file-existence verification at the expected path — is a plan-phase decision. If no mechanism allows confirmation at log time, FR-011's "neutral wording when outcome unknown" path applies: the log line uses "save requested" wording without success claims, and FR-012 (path in success log) MAY be deferred to whenever confirmation does become available (e.g., via the OBS muxer's own write log line that the user can correlate).
- Sanitization of the slot name applies only to the constructed replay filename (FR-004a). The continuous recording filename construction at `src/slot.cpp:96-103` is unchanged per FR-006/FR-008; it continues to embed the raw `cfg.name` value. The resulting asymmetry — a slot whose user-facing name contains path-illegal characters will have its recording fail at write time (surfaced via `obs_output_get_last_error` at the existing recording call sites) while its replays succeed with a sanitized name — is an accepted consequence of the "no change to the recording filename pattern" boundary. Bringing recording sanitization to parity is a separate-feature concern.
- Whether OBS's strftime-style format-string substitution (`%CCYY`, `%MM`, `%DD`, `%hh`, `%mm`, `%ss`) tolerates underscore-delimited literal text containing arbitrary sanitized slot-name and id-prefix characters in every locale OBS supports is a plan-phase investigation. If a locale is found where some sanitized character sequence interacts with the format-string parser, the sanitization character set MUST be tightened to exclude that sequence; the spec-level guarantee (uniqueness + readability + truthful logging) is unaffected.
