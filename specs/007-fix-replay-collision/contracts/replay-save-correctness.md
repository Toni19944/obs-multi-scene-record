# Contract — Replay-Save Correctness

**Feature**: Replay file uniqueness across slots sharing an output directory + truthful replay-save logging

**Branch**: `007-fix-replay-collision` | **Date**: 2026-05-25

This contract is the single source of truth for two coupled invariants this feature establishes:

1. **Filename uniqueness invariant**: no two slots in a single configuration can produce identical replay file names in the same directory under any concurrent-save pattern.
2. **Log truthfulness invariant**: the plugin's log line for a replay save never claims success unless the plugin has confirmed a file was written.

The two invariants are documented together because the same code path (`SceneSlot::setup_outputs` + `SceneSlot::save_replay` + the new `saved` signal callback) carries both.

---

## Surfaces in scope

| Surface | Today (pre-007) | After 007 |
|---|---|---|
| Replay-buffer output `"format"` setting | Literal `"Replay_%CCYY-%MM-%DD_%hh-%mm-%ss"` | `"<sanitized-name>_<id6>_Replay_%CCYY-%MM-%DD_%hh-%mm-%ss"` |
| Plugin log line at proc-dispatch site | `'<slot>' replay save OK` (INFO) on `true`, `'<slot>' replay save FAILED` (INFO, sic) on `false` | `'<slot>' replay save requested` (INFO) on `true`, `'<slot>' replay save proc-dispatch FAILED (slot not capturing?)` (WARNING) on `false` |
| Plugin log line at mux-completion | (none — no subscription to `saved`) | `'<slot>' replay save wrote '<path>'` (INFO) on every successful save |

## Filename uniqueness invariant

### Statement

For any two SceneSlots `A` and `B` belonging to the same `SlotManager` (same scene-collection load), `build_replay_format(A.cfg) != build_replay_format(B.cfg)`.

### Proof sketch

- `cfg.id` is generated by `generate_slot_id()` (`slot.cpp:61-69`) once at slot creation. The combination of `os_gettime_ns()` (high resolution) and a process-lifetime atomic counter guarantees `A.id != B.id` for any two slots created in the same process.
- The last 6 hex characters of an id consist of (low bits of ns) ⊕ (low bits of counter) — different across rapid successive creations because the counter increments. (See [research.md](./research.md) D3 for the choice rationale.)
- The format string is `"<sanitized-name>_<id6>_Replay_..."`. Since `<id6>` is positionally fixed and always present, two slots with `A.id != B.id` produce two distinct format strings, regardless of whether `sanitize(A.name) == sanitize(B.name)`.

### Resulting on-disk filenames

OBS's `os_generate_formatted_filename` composes the final filename as `<directory>/<format>.<extension>`. The `<directory>` and `<extension>` keys can be the same across slots (this is the configuration the bug occurs in); the `<format>` key — set per-slot from the helper — differs by `<id6>` at minimum. Therefore the full on-disk paths differ.

### Edge cases satisfied by the invariant

| Case | How the invariant holds |
|---|---|
| Two slots same dir, same container, same wall-clock second save | `<id6>` differs → distinct filenames. |
| N slots same dir, same wall-clock second save | All N `<id6>` values differ → all N filenames distinct. |
| Two slots share `cfg.name` (slot names are not enforced unique) | `<id6>` differs → distinct filenames. |
| One slot's `cfg.name` is empty | Fallback `"slot"` literal; `<id6>` still differs from other slots' `<id6>` → distinct. |
| Slot name contains only sanitized-away characters | Sanitization yields empty → fallback `"slot"` literal; `<id6>` still differs → distinct. |
| Slot renamed at some point | The rename affects `cfg.name` but not `cfg.id`; `<id6>` is stable → uniqueness holds across renames. Previously-saved files keep their old filenames (FR-009). |
| Slot name differs only by case ("CamA" vs "camA") | `<id6>` differs → distinct, independent of filesystem case sensitivity. |

### Non-goals of the invariant

- It does NOT guarantee uniqueness across different `SlotManager` instances (e.g., the same plugin loaded twice into two OBS instances both writing to a network drive). Out of scope.
- It does NOT guarantee uniqueness against pre-007 replay files that already exist in the directory. Pre-007 files use the literal `Replay_<ts>` format; post-007 files use `<name>_<id6>_Replay_<ts>`. These can never collide (the post-007 shape has at least one extra `_<id6>_` segment that pre-007 lacks).
- It does NOT guarantee uniqueness against arbitrary user-placed files in the same directory whose names happen to match a post-007 filename. (File-creation-with-overwrite-prevention is the OS's responsibility; libobs would fail the write if it occurred. That failure would surface via the absence of the `wrote` log line.)

## Log truthfulness invariant

### Statement

For any Save Replay operation initiated by `save_replay()`:

- If `proc_handler_call(ph, "save", &cd)` returns `true` AND the OBS mux thread later emits the `"saved"` signal, exactly one `wrote '<path>'` log line is emitted via the callback with `<path>` set to the `get_last_replay`-reported on-disk path. SUCCESS WORDING APPEARS.
- If `proc_handler_call(ph, "save", &cd)` returns `true` BUT the OBS mux thread does NOT emit the `"saved"` signal (because the mux thread hit the `error = true` path at `mux.c:1099` / `1103` / `1112`), NO `wrote` log line is emitted. NO SUCCESS WORDING APPEARS.
- If `proc_handler_call(ph, "save", &cd)` returns `false`, the `request` line uses LOG_WARNING wording (`proc-dispatch FAILED`). NO SUCCESS WORDING APPEARS.

### Cases satisfying the invariant

| Cause of save failure | `proc` return | `saved` signal | Plugin log output | Truthful? |
|---|---|---|---|---|
| Slot not capturing (replay buffer not started) | `false` | (n/a) | `request: proc-dispatch FAILED` | ✓ |
| Cross-slot filename collision (pre-007 only — fixed structurally by this feature) | `true` | not emitted (`mux.c:error=true` path) | `request` only, no `wrote` | ✓ |
| Disk full mid-write | `true` | not emitted (write_packet returns false → error path) | `request` only, no `wrote` | ✓ |
| Output directory missing or unwritable at save time | `true` | not emitted (start_pipe fails → error path) | `request` only, no `wrote` | ✓ |
| Permission revoked between save dispatch and mux thread starting | `true` | not emitted (start_pipe fails) | `request` only, no `wrote` | ✓ |
| Successful save | `true` | emitted | `request` then `wrote '<path>'` | ✓ |

### Agreement with the OBS muxer's own log

Per FR-013, the plugin's `wrote` line agrees with the OBS muxer's `Wrote replay buffer to '<path>'` line for the same save:

- Both are emitted from `replay_buffer_mux_thread` only on the `!error` path (the OBS line at `mux.c:1118`, the plugin's at `on_replay_saved` triggered by the signal at `mux.c:1133`).
- The plugin's `<path>` is obtained via `get_last_replay`, which returns `stream->path.array` at `mux.c:947` — the exact string variable the OBS log line at `mux.c:1118` reads.
- The two log lines are necessarily for the same save event because both are tied to the same mux thread invocation.

If the user ever observes a plugin `wrote` line whose path disagrees with the OBS `Wrote replay buffer to` line for the same save, that is a regression bug per this contract.

## Threading contract

### Lock order (unchanged from constitution Principle III)

```
SlotManager::mtx_ → SceneSlot::slot_mtx_ → SceneSlot::stats_mtx_ → SlotManager::shared_mtx_ (leaf)
```

No new mutexes. No new locks acquired by code touched in this feature.

### Per-method lock state

| Method | Thread | Lock state |
|---|---|---|
| `replay_util::build_replay_format(cfg)` | Caller's thread | None (pure function) |
| `setup_outputs` (which calls `build_replay_format` and `signal_handler_connect("saved")`) | UI thread (during `start()`) | `slot_mtx_` (held by `start()`) |
| `save_replay` (proc dispatch + `request` log line) | UI / hotkey / caller's thread | `slot_mtx_` (entered at `slot.cpp:1056`) |
| `on_replay_saved` (signal callback — static C-style) | OBS muxer worker thread | **No plugin locks** |
| `log_replay_saved` (instance method called by callback) | OBS muxer worker thread | **No plugin locks** |
| `teardown_locked` (disconnect + stop + wait + release) | UI thread (during `stop()`) | `slot_mtx_` (held by `stop()`) |

### Synchronization barrier

`signal_handler_disconnect("saved", ...)` in `teardown_locked` is the sole synchronization point between the mux-thread callback and the UI-thread teardown. libobs's `signal.c` serializes signal dispatch and disconnect on the signal's internal mutex; the disconnect call blocks until any in-flight callback returns. **After the disconnect call returns, the callback is guaranteed not running and cannot fire again.**

This guarantees:

1. `obs_output_release(replay_out_)` at `slot.cpp:683` runs after the callback has completed — the callback can never observe a stale or freed `replay_out_`.
2. `replay_out_ = nullptr` at `slot.cpp:683` runs after the callback has completed — the callback can never observe a null `replay_out_` *during* its execution (it might be entered briefly with a non-null value while the disconnect is processing, but it completes before the disconnect returns, so the assignment to nullptr happens after the callback's read).

### Forbidden contexts for the callback

The `on_replay_saved` callback MUST NOT:

- Acquire `slot_mtx_`, `mtx_`, `stats_mtx_`, or `shared_mtx_` (would deadlock with `teardown_locked`'s disconnect-while-holding-`slot_mtx_`).
- Call `self->stop()`, `self->teardown()`, or any other method that acquires `slot_mtx_`.
- Touch Qt widgets directly (Constitution Principle IV; UI updates from worker threads must use `QMetaObject::invokeMethod` queued connections, the same pattern as `slot.cpp:1006-1007`).
- Block on I/O. (The log emission via `blog` is fast and non-blocking enough; `proc_handler_call("get_last_replay")` is a single accessor inside the muxer, also fast.)

### Permitted contexts

The callback MAY:

- Read `self->cfg_.name` lock-free (mild race with editor edits, tolerated per project convention at `slot.cpp:1037-1038`).
- Read `self->replay_out_` lock-free (safe per the disconnect barrier above).
- Call `obs_output_get_proc_handler(self->replay_out_)` and `proc_handler_call(ph, "get_last_replay", ...)`.
- Read `calldata_string(&cd, "path")`.
- Call `blog(LOG_INFO, ...)`.

They have been part of OBS's public replay-buffer API since the replay-buffer feature was introduced in OBS Studio 19.0 (2017) — well below the `buildspec.json` floor of OBS 31.1.1.
