# Feature Specification: Replay buffer honors configured duration under quality-based rate control

**Feature Branch**: `008-fix-replay-quality-truncation`

**Created**: 2026-05-25

**Status**: Draft

**Input**: User description: "Replay buffer truncates to random short durations when rate control is quality-based"

## Clarifications

### Session 2026-05-25

- Q: Should the slot editor expose a user-facing "Max replay buffer size (MB)" control, in addition to the per-mode auto-derived ceiling? → A: Auto-derived with writable override. The plugin computes a default ceiling per FR-001 / FR-003 / FR-004 (deterministic from configuration). The slot editor exposes a per-slot "Max replay buffer size (MB)" spinbox; when empty / 0 / unset, the auto-derived value is used; when set to a positive value, the user-supplied value is used as-is and replaces the auto-derived ceiling. The currently resolved ceiling (auto-derived or user-supplied) is visible in the editor so the user can see the memory cost before starting. This expands the spec's editor surface beyond what was originally "out of scope" and introduces one new persisted per-slot field.
- Q: Should in-session runtime re-estimation of the replay buffer ceiling from observed bitrate be in scope? → A: Out of scope; measure for logging only. The ceiling stays deterministic from configuration per FR-004 — no stop/restart of the replay-buffer output to grow it mid-session (that operation would discard the buffered window, defeating its purpose). The plugin MAY sample observed bitrate (e.g., via `obs_output_get_total_bytes` deltas) and surface it in the save log and slot start log as diagnostic information — see FR-014 — so the user can correlate "auto-derived ceiling assumed X Mbps; actual sustained Y Mbps" and decide whether to set the FR-012 override on the next start. The FR-012 user-facing override is the supported escape valve when auto-derivation under-shoots; runtime resize is not.
- Q: How specific should the spec be about the auto-derived sizing formula? → A: Lock the structural shape; defer specific numbers to the plan. The spec binds (i) per-mode branches — separate branches for bitrate-based, quality-based, and lossless modes — so the plan cannot regress to a single flat constant; (ii) safety margin of at least 2× to cover VBR peaks; (iii) the quality-mode branch's estimate MUST scale with at least resolution × fps (a per-pixel-per-frame heuristic), NOT be a flat Mbps constant, so a single coefficient covers 1080p60 and scales correctly to 1440p144 / 4K60 without spec amendment; (iv) the lossless branch gets its own much higher estimate independent of the quality-mode branch; (v) the existing audio bitrate × track count addition is preserved. The plan picks the specific coefficients and verifies against the maintainer's reference content (a known-demanding case: CQP-17 at 1080p60 with NVENC P5/full-res-double-pass/high-quality on a modern open-world third-person shooter, observed peaking at 60+ Mbps). The FR-012 user override is the safety valve for configurations the default still under-shoots — matching the OBS-native replay-buffer model of "sensible auto-derived default + user-editable override."
- Q: When the requested `max_size_mb` (auto-derived OR user-supplied via the FR-012 override) cannot be allocated at slot start, what should happen? → A: Clamp-and-warn. The plugin allocates whatever the host can spare for the replay buffer, starts the buffer at the smaller capacity, and emits a warning log line that the configured duration will NOT be honored at this ceiling — naming the slot, the requested ceiling, the clamped ceiling actually used, and pointing the user at the FR-012 override or other configuration knobs (reduce `replay_seconds`, lower the quality setting, switch to a bitrate-based mode) as the actionable next step. This applies symmetrically to the auto-derived ceiling and the user-supplied FR-012 override. Refusing the slot entirely was rejected because it loses recording too; declining only the replay buffer (no clamp) was rejected because "some replay buffer with a loud warning" was preferred over "no replay buffer at all." US1's duration contract is explicitly documented as suspended for the clamped case — see FR-006 and FR-011.
- Q: How should the plugin detect that the memory cap caused a short replay save (FR-011 trigger)? → A: Bitrate inference from runtime stats, hedged wording. The /speckit-plan API audit (research.md D4) found that libobs does not currently expose a cause-signal at save time (no `replay_buffer_get_current_bytes` or equivalent introspection of the buffer's current-bytes vs `max_size_mb`). The implementation therefore uses an **EWMA bitrate inference**: at save time, compare the EWMA-smoothed observed kbps × `replay_seconds` against the resolved `max_size_mb` — when the inferred buffer-need exceeds the resolved cap, emit a hedged warning naming the observed bitrate, the auto-derived ceiling's assumed bitrate, the inferred need in MB, and the resolved cap — phrased as "likely truncated; suspected memory cap (cause not directly confirmed)" rather than asserting the cause. The wording is hedged because this is a *prediction* from observed bitrate, not a *measurement* of the saved file's duration (the file's actual duration is not probed; no portable OBS API, and the cost of reading the saved file back to measure its duration outweighs the signal value). Three suppression conditions narrow the trigger to prevent false-positive log spam: (1) slot wall-clock uptime since `start()` ≥ `replay_seconds` — the buffer had time to fill; (2) the FR-006 clamp path did not already warn at slot start — its warning is not repeated per-save; (3) the EWMA has accumulated at least one sample. If OBS later exposes a clean cause-signal at save time, the inference path MAY be replaced or augmented and the wording MAY upgrade from "suspected" to "confirmed."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Saved replay length matches the configured replay length under quality-based rate control (Priority: P1)

A user configures a slot for quality-based rate control (CQP, CRF, ICQ, CQ, QVBR, or any other non-bitrate-targeted mode) — the natural choice when the user prioritizes visual quality and is willing to accept variable file size. The slot's replay buffer is enabled with a user-chosen duration (for example, "keep the last 30 seconds"). The user triggers Save Replay during typical gameplay or screen capture activity.

Today, the saved replay file frequently contains far less than the configured duration — sometimes a few seconds, sometimes ten or twenty, varying from save to save and varying with what was happening on screen at the time. The user has no way to predict how much footage they will actually get. High-motion or visually complex content gets the shortest clips; static content gets closer to the requested length. The user-configured "30 seconds" reads as advisory rather than a contract.

After this fix, a Save Replay against a slot configured for N seconds MUST produce a clip approximately N seconds long (within encoder GOP / frame-boundary quantization), regardless of the rate-control mode in effect and regardless of the scene complexity during the buffered window. The configured duration is the contract.

**Why this priority**: P1 because the user-configured replay duration is the single most user-visible setting on the replay buffer feature, and it currently silently does not honor the configured value when the rate-control mode is quality-based. The user reasonably expects "30 seconds" to mean "30 seconds." This is data loss in slow motion: the user gets a clip that ends before the moment they meant to capture began, with no signal that anything went wrong.

**Independent Test**: Configure one slot for a quality-based mode at a setting that produces a high bitrate (for example, CQP 18 at 1080p60 on a hardware encoder; or CRF 18 on `obs_x264`). Set the replay duration to 30 seconds. Start the slot, point a high-motion source at it (a game with fast camera movement, a video playback, a screen recording with rapid window scrolling) and wait at least 30 seconds. Trigger Save Replay. Inspect the saved file's duration. The duration MUST be approximately 30 seconds — not 4 seconds, not 12 seconds, not "whatever fit." Repeat across multiple saves under varying scene complexity within the same session; every save MUST produce approximately the same duration.

**Acceptance Scenarios**:

1. **Given** a slot configured for a quality-based rate-control mode at a typical high-quality setting, and a replay buffer length of N seconds, **When** the user triggers Save Replay after the buffer has been running at least N seconds against high-complexity content, **Then** the saved file's duration is approximately N seconds (within one keyframe interval of the configured value).
2. **Given** the same configuration, **When** the user triggers Save Replay against low-complexity content (mostly static screen, idle desktop), **Then** the saved file's duration is also approximately N seconds — the duration does NOT vary with scene complexity.
3. **Given** two consecutive Save Replay triggers against the same slot during the same session with content of differing complexity, **When** both files are inspected, **Then** their durations are approximately equal (both at approximately N seconds), with no "this one is shorter because there was more motion" outcome.
4. **Given** a slot configured for a bitrate-based rate-control mode (CBR, VBR, ABR), **When** the user triggers Save Replay under any scene-complexity condition, **Then** the existing duration-honoring behavior for bitrate modes is preserved — this fix MUST NOT regress the bitrate-mode case (which already works because the buffer's memory cap is computed from the user-set bitrate, which the encoder respects).

### User Story 2 - Per-slot memory use of the replay buffer is bounded and predictable (Priority: P2)

A user runs several slots simultaneously, several of them with the replay buffer enabled at long durations and high-quality rate-control modes. The user's host machine has finite RAM. The user must be able to trust that enabling replay buffers will not allocate unbounded memory per slot when quality-based rate control happens to produce extreme bitrates (4K60 high-motion content under CQP 14 can momentarily exceed 200 Mbps).

After this fix, while the replay buffer reliably holds the configured duration (US1), the per-slot memory commitment for the replay buffer MUST remain bounded by a deterministic function of the user's configuration (resolution, fps, replay duration, container, rate-control settings) — not by the runtime peak bitrate of an unpredictably complex scene. The user MUST be able to reason about "I have 4 slots at 30 seconds each at this resolution; that will use approximately X MB." The fix MUST NOT trade US1's "configured duration is honored" guarantee for an "and it will use however much RAM the moment demands" footnote.

**Why this priority**: P2 because the maintainer is the only user reachable through the current distribution channel (a small-team / self-hosted plugin, not a mass-market product), so out-of-memory crashes from a too-generous buffer estimate are a real concern on the maintainer's actual hardware — but the primary user-visible bug is US1, and US2 is a constraint on the US1 solution rather than a separate user goal. A US1 implementation that satisfies the duration contract but allocates 2 GB per slot in the worst case is unacceptable; a US1 implementation that satisfies the duration contract with a deterministic per-slot ceiling derived from configuration is the target.

**Independent Test**: Configure four slots, each with a quality-based rate-control mode at a typical high-quality setting, each with a 30-second replay buffer, all at the same resolution and fps. Start all four. Monitor the plugin's reported per-slot replay-buffer memory commitment (existing log line at slot start, or process resident-set growth attributable to the buffers). The total replay-buffer memory MUST be bounded by approximately 4 × (the per-slot ceiling derived from the configuration), regardless of how much motion is on screen during the test. Increase scene complexity to the maximum the host can encode; the memory cap MUST NOT exceed the configured ceiling by more than a small safety margin.

**Acceptance Scenarios**:

1. **Given** N slots each with the replay buffer enabled at the same configuration (resolution, fps, duration, rate-control mode), **When** they are all running under varying scene complexity, **Then** total replay-buffer memory commitment is bounded by approximately N × (a deterministic per-slot ceiling that the user could have predicted from the configuration alone).
2. **Given** a slot with the replay buffer enabled at extreme settings (4K60 at the highest quality setting the encoder supports, longest user-configurable duration), **When** the slot starts, **Then** the per-slot ceiling MUST be reported in a way the user can observe (existing slot start log line, or a dedicated "replay buffer reserved N MB" line), so the user can confirm the commitment matches their RAM budget before complaining.

### User Story 3 - User can tell from the plugin log whether the replay buffer's memory ceiling caused early eviction (Priority: P3)

In the cases where a user's chosen rate-control settings produce a runtime bitrate that exceeds even the corrected estimate (US1) AND the user has not increased the per-slot ceiling, the buffer may still evict older frames before reaching the configured duration. The user MUST be able to tell from the plugin's log alone whether this happened, so they can attribute a too-short clip to a memory-cap eviction rather than a different bug.

After this fix, when the plugin's runtime stats indicate the buffer's memory ceiling likely caused early eviction (the EWMA-smoothed observed bitrate × `replay_seconds` exceeds the resolved `max_size_mb`, with uptime ≥ `replay_seconds` and no prior FR-006 clamp warning at slot start), the plugin's log line for that save MUST emit a hedged warning naming the configured duration, the observed bitrate, the auto-derived ceiling's assumed bitrate, the inferred buffer-need in MB, and the resolved cap in MB — for example, "configured 30s; observed 45 Mbps suggests buffer needed ~169 MB but resolved cap is 87 MB (assumed 12 Mbps); suspected memory cap (cause not directly confirmed)." The wording is hedged because this is a bitrate-inference prediction, not a measurement of the saved file's actual duration (no portable OBS API for reading the saved file back; research.md D4). This is an observability requirement that complements the truthful-save logging from feature 007 (its FR-011 / FR-012 / FR-013 — not to be confused with this spec's FR-011 / FR-012 / FR-013, which have different meanings).

**Why this priority**: P3 because the US1 fix is expected to eliminate this case for typical configurations; this user story covers the residual edge case where the user has chosen a rate-control configuration that produces bursts beyond the per-slot ceiling US2 places. Without US3, that residual case is observationally indistinguishable from a plugin bug. With US3, the user knows to either raise the ceiling, reduce the duration, or lower the quality setting.

**Independent Test**: Force the residual case — configure a slot whose runtime EWMA-observed bitrate × `replay_seconds` will exceed the resolved cap (e.g., set the FR-012 override to a value smaller than the auto-derived ceiling and run a high-bitrate scene long enough for the EWMA to settle above the cap's assumed kbps). Wait until the slot's wall-clock uptime exceeds `replay_seconds`. Trigger Save Replay. Inspect the plugin log line for that save. The line MUST identify (a) the configured `replay_seconds`, (b) the EWMA-observed bitrate, (c) the auto-derived ceiling's assumed bitrate, (d) the inferred buffer-need in MB, and (e) the resolved cap in MB, hedged with "suspected memory cap (cause not directly confirmed)" wording. Compare with a Save Replay under normal conditions where the observed bitrate stays below the cap's assumed bitrate; the line MUST NOT include the cap-related warning for that case (per FR-010).

**Acceptance Scenarios**:

1. **Given** a Save Replay completed and the resulting file's duration is approximately equal to the configured duration, **When** the plugin emits its log line for that save, **Then** the line does NOT include any "memory cap reached" or "buffer truncated" wording — only the truthful-success wording from feature 007's FR-012.
2. **Given** a Save Replay completed against a slot whose runtime EWMA-observed bitrate × `replay_seconds` exceeds the resolved `max_size_mb` (and uptime ≥ `replay_seconds`, and the FR-006 clamp path did not already warn at slot start), **When** the plugin emits its log line for that save, **Then** the line emits the hedged warning identifying the configured `replay_seconds`, the observed bitrate, the auto-derived ceiling's assumed bitrate, the inferred buffer-need in MB, and the resolved cap in MB, phrased as "suspected memory cap (cause not directly confirmed)" rather than asserting the cause.
3. **Given** a Save Replay failed for any reason unrelated to the memory cap (write error, missing directory, encoder stopped), **When** the plugin emits its log line for that save, **Then** feature 007's FR-011 wording rules apply — the line does NOT misattribute the failure to a memory-cap eviction.

### Edge Cases

- **Slot uses a shared (consumer) encoder whose owner is configured for quality-based rate control**: Feature 006's FR-014 already requires the consumer's replay-buffer memory-cap estimate to read the owner's effective rate-control mode and value, not the consumer's stale fields. This spec inherits that wiring: the "quality mode → corrected sizing" path described in FR-001/FR-002 below applies when the *effective* mode (owner's) is quality-based, regardless of whether the consumer's own stored fields say otherwise. No new resolution code is introduced; the existing `eff` value at the existing call site is the input.
- **Encoder construction falls back to `obs_x264` + CBR at 6000 kbps** (existing fallback at `src/slot.cpp:430` when the requested encoder is unavailable): After fallback, the effective mode is bitrate-based and the effective value is 6000 kbps. The buffer is sized from those effective values per the existing bitrate-based path. This fix MUST NOT alter the fallback case (it is already correctly sized at the new effective bitrate).
- **Lossless rate-control mode**: Lossless encoding produces by far the highest bitrates of any mode and is the worst case for the bug this spec fixes. The per-slot ceiling for the lossless case may be substantially larger than for typical quality-mode settings; this is acceptable as long as the ceiling is bounded by a deterministic function of resolution/fps/duration and is observable to the user per US2 acceptance scenario 2.
- **User configures an extreme replay duration (e.g., 600 seconds) with a high-bitrate quality mode at 4K60**: The deterministic per-slot ceiling implied by that configuration may exceed available host RAM. The fix MUST NOT crash or OOM the host as a way of honoring US1; per FR-006's clamp-and-warn behavior (resolved from clarification Q4), the plugin allocates the largest ceiling the host can provide, starts the replay buffer at that clamped ceiling, and warns that the configured `replay_seconds` will NOT be honored — the user sees the clamped vs requested numbers in the log and decides which knob to turn (FR-012 override, lower `replay_seconds`, lower quality, switch to a bitrate-based mode). The rest of the slot (recording, other slots) MUST continue independently.
- **Multiple replay-enabled slots whose combined ceilings exceed host RAM**: Each slot's pre-emptive sizing check (FR-006) runs independently against `available_physical_mb()` at that slot's start time. Because the OBS replay-buffer output does NOT pre-allocate the ring buffer (frames grow it lazily over wall-clock time), earlier slots' buffers don't immediately reduce the available-RAM reading — the reading drops over time as those buffers fill. Clamp severity for a given slot therefore depends on its start order and on what earlier slots' buffers have already grown to. Any slot whose pre-emptive check would request more than the FR-006 fraction of currently-available RAM goes through clamp-and-warn — host fragmentation and other processes' memory use may also reduce the threshold. Each clamped slot logs its own request-vs-clamp values via FR-006's warning line. Individual slot-start clamps MUST be local (one slot's clamp MUST NOT prevent the others from starting).
- **Audio track count and audio bitrate**: Audio data also lives in the replay-buffer ring (existing code already adds `audio_bitrate × popcount(audio_tracks)` to `est_kbps` at `src/slot.cpp:898`). The fix MUST preserve this addition under the new quality-mode sizing path — the per-slot ceiling MUST account for the same audio overhead the existing bitrate-mode path accounts for.
- **Container format affects mux overhead**: Different container formats (MKV, MP4, fragmented MP4) have different per-frame mux overhead; the existing code's 1.5× safety multiplier (`est_kbps * cfg_.replay_seconds / 8 / 1024 * 3 / 2` at `src/slot.cpp:905`) provides headroom for this. The fix MUST preserve a comparable safety margin under the new quality-mode sizing path.
- **Existing 50 MB floor** (current `if (max_size_mb < 50) max_size_mb = 50;` at `src/slot.cpp:906`): This floor was sized for the bitrate-mode default of 12 Mbps with very short replay windows. Under the new quality-mode sizing, the computed ceiling will routinely exceed 50 MB; the floor remains useful only as a defensive lower bound for very short replay windows on low-bitrate bitrate modes. The fix MAY preserve or adjust this floor; either choice is acceptable as long as the US1 duration contract holds. **Semantic note**: the implementation repurposes the 50 MB constant from a *raise-to-floor* (always at least 50 MB) into a *decline cutoff* (decline the replay buffer entirely when the post-clamp ceiling falls below 50 MB) per FR-006. The "always ≥50 MB" guarantee for the unclamped path is therefore retired — small computed ceilings on low-bitrate / short-window configurations are passed to OBS as-is rather than silently raised. This is intentional and within the "MAY preserve or adjust" latitude this edge case grants.
- **Configured replay duration of zero or negative**: Defensive handling already exists upstream (the editor clamps the value); this fix does NOT introduce new validation, but the buffer-sizing math MUST NOT divide by zero, overflow, or produce a negative `max_size_mb` for any non-negative `replay_seconds`.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: When a slot's effective rate-control mode is quality-based (CQP, CRF, ICQ, CQ, QVBR, or any other non-bitrate-targeted, non-lossless mode in the `rc_util::is_bitrate_based` sense at `src/slot.cpp:22-30`), the replay buffer's memory ceiling MUST be sized to hold the user-configured `replay_seconds` worth of video produced by that encoder configuration under typical worst-case scene complexity for the configured resolution and fps. The estimate MUST scale with at least resolution × fps (a per-pixel-per-frame heuristic), NOT a flat Mbps constant, so the same coefficient applies across 1080p60 and higher resolutions / framerates. The current hardcoded "assume 12 Mbps" fallback at `src/slot.cpp:891` MUST be replaced. The lossless rate-control mode is handled by FR-001a; this requirement covers the non-lossless quality-based modes.
- **FR-001a**: When a slot's effective rate-control mode is lossless (`rc_util::is_lossless(mode)` at `src/slot.cpp:32-35`), the replay buffer's memory ceiling MUST be sized from a separate, much higher per-pixel-per-frame estimate than FR-001's quality-mode coefficient — lossless H.264 / HEVC at 1080p60 routinely produces an order of magnitude more bitrate than CQP at typical quality settings. The estimate MAY be derived from the raw frame size (resolution × bits-per-pixel of the encoder's input format) × fps × a compression-ratio assumption, or from a separate flat coefficient; the specific approach is plan-phase. The lossless branch MUST be distinct from the FR-001 quality-mode branch — they MUST NOT share a single coefficient.
- **FR-001b**: The auto-derived ceiling MUST apply a safety margin of at least 2× to the per-mode estimate (raised from the current 1.5×) to cover VBR peaks that exceed sustained-bitrate assumptions.
- **FR-002**: For a saved replay against a slot configured for N seconds and a quality-based rate-control mode, the saved file's duration MUST be approximately N seconds (within one keyframe / GOP interval of the configured value, the same tolerance OBS's replay buffer applies for bitrate-based modes today). The saved duration MUST NOT vary with scene complexity during the buffered window.
- **FR-003**: The fix MUST NOT regress the bitrate-mode sizing path. For slots whose effective rate-control mode is bitrate-based, the existing memory-cap derivation (user-set kbps × configured seconds / 8 with the existing 1.5× safety multiplier) MUST continue to apply unchanged; the new quality-mode sizing path MUST be a separate branch, not a replacement of the shared path.
- **FR-004**: The per-slot memory ceiling derived from the user's configuration (resolution, fps, replay duration, rate-control mode and value, audio configuration, container) MUST be a deterministic function of those inputs. Identical configurations MUST produce identical ceilings; the ceiling MUST NOT depend on runtime observations of actual produced bitrate. The user MUST be able to predict per-slot memory commitment from the configuration alone.
- **FR-005**: The per-slot memory ceiling for the replay buffer MUST be reported in a form the user can observe at slot start time — either by extending the existing slot start log line or by adding a dedicated "replay buffer reserved N MB" line. This MUST occur for every slot start that enables the replay buffer, regardless of rate-control mode (bitrate-based slots also report their ceiling, for parity).
- **FR-006**: At slot start, the plugin MUST pre-emptively size the requested `max_size_mb` against the host-reported available physical RAM at that moment (probed via `available_physical_mb()` — platform helper per research.md D3). When the computed ceiling (auto-derived per FR-001 / FR-001a / FR-001b, OR user-supplied via the FR-012 override) exceeds a plan-chosen fraction of `available_physical_mb()` (plan-phase choice: half, leaving headroom for OBS, other plugin slots, and other processes), the behavior MUST be **clamp-and-warn**: the plugin clamps the requested value to that fraction, passes the clamped value to the OBS replay-buffer output as `max_size_mb`, and emits a warning log line identifying (a) the slot, (b) the requested ceiling, (c) the clamped ceiling actually used, (d) that the configured `replay_seconds` will NOT be honored at this ceiling, and (e) the actionable next steps (set or lower the FR-012 override, reduce `replay_seconds`, lower the quality setting, or switch to a bitrate-based rate-control mode). The slot's other outputs (continuous recording, if configured) MUST start normally; other slots' replay buffers MUST start independently. When the clamp falls below the 50 MB defensive floor (plan-phase choice), the replay buffer for that slot MUST be declined (not started), with an error log line in the same shape as the clamp warning. The slot start MUST NOT crash the application and MUST NOT silently honor the smaller ceiling without telling the user.

  **Scope**: FR-006 is **pre-emptive sizing only**. The OBS replay-buffer output does NOT pre-allocate the ring buffer at output start — it stores `max_size_mb` as an upper bound and grows the buffer lazily as frames arrive (research.md D3). FR-006 therefore covers only the host-RAM precondition that can be observed pre-emptively at slot start via the available-physical-RAM probe. Mid-stream allocation pressure — the ring buffer hitting its cap during runtime and evicting frames — is detected and surfaced by FR-011's bitrate-inference path, not by this requirement. Generic slot-start errors unrelated to RAM (encoder unavailable, codec failure, container construction error) are handled by the plugin's existing slot-start error path and are out of scope for FR-006.
- **FR-007**: For consumer slots (slots whose video encoder is shared from an owner slot), the rate-control mode and value used to size the replay buffer MUST be the owner's effective values, exactly as feature 006's FR-014 already requires for the existing call site at `src/slot.cpp:891` (which passes `eff.mode` / `eff.value`). This spec MUST NOT introduce a new resolution path; it MUST consume the same `eff` value the existing code consumes and apply the new quality-mode sizing on top of it.
- **FR-008**: The fix MUST NOT alter the shape, default value, or meaning of any existing user-configurable persisted slot setting — replay duration, rate-control mode, rate-control value, output directory, container, etc. One new persisted per-slot field IS added by FR-012 (the "Max replay buffer size (MB)" override, resolved from clarification Q1); existing fields are untouched. Beyond adding that one field, the change is confined to the runtime computation of `max_size_mb` passed to the replay-buffer output's settings.
- **FR-009**: The fix MUST NOT alter the existing `max_time_sec` setting passed to the replay-buffer output (`obs_data_set_int(rb, "max_time_sec", cfg_.replay_seconds)` at `src/slot.cpp:904`). The configured duration remains the time bound; this fix ensures the memory bound is generous enough that the time bound is the binding constraint under typical conditions.
- **FR-010**: When a Save Replay completes and the saved file's duration is approximately equal to the configured duration, the plugin's log line for that save MUST NOT include any wording suggesting the buffer's memory cap influenced the result. (This complements feature 007's FR-012 truthful-success wording.)
- **FR-011**: At save time, the plugin MUST compute a bitrate-inference check: take the EWMA-smoothed observed kbps (FR-014), multiply by the configured `replay_seconds`, divide by 8 / 1024 to get an inferred buffer-need in MB, and compare against the resolved `max_size_mb`. When the inferred need exceeds the resolved cap AND all three suppression conditions are clear — (1) slot wall-clock uptime since `start()` ≥ `replay_seconds` (the buffer had time to fill; short-after-start is not a cap event), (2) the FR-006 clamp path did not already warn at slot start (that case has its own warning; repeating it per-save is log spam), (3) the EWMA has accumulated at least one sample — the plugin's log line for that save MUST emit a hedged warning identifying (a) the configured `replay_seconds`, (b) the EWMA-observed bitrate, (c) the auto-derived ceiling's assumed bitrate, (d) the inferred buffer-need in MB, and (e) the resolved cap in MB, phrased as "likely truncated; suspected memory cap (cause not directly confirmed)" rather than asserting the cause. The wording is hedged because this is a bitrate-inference *prediction*, not a *measurement* of the saved file's actual duration — the implementation does not probe the saved file's duration (no portable OBS API for reading the saved file's duration back; research.md D4). This MUST NOT be confused with feature 007's FR-011 failure wording — the save itself succeeded and produced a file; this warning is about the file's likely duration vs configured, not about save success/failure. If OBS later exposes a clean cause-signal at save time (e.g., the buffer's current-bytes vs `max_size_mb` at save), the inference path MAY be replaced or augmented and the wording MAY upgrade from "suspected" to "confirmed."
- **FR-012**: The slot editor MUST expose a per-slot writable "Max replay buffer size (MB)" control. When the control is empty / 0 / unset, the plugin MUST use the auto-derived ceiling from FR-001 / FR-003 / FR-004 as the value of `max_size_mb` passed to the replay-buffer output. When the control is set to a positive integer, the plugin MUST use that exact value as `max_size_mb` (the auto-derived value is computed and shown but not applied). The editor MUST display the currently resolved ceiling — auto-derived or user-supplied — in a form the user can see before starting the slot (e.g., as the spinbox's placeholder text when empty, or as an adjacent "resolved: N MB" label). The override MUST be persisted as one new per-slot configuration field; existing pre-feature slot configurations MUST load with the override unset (i.e., auto-derived behavior, backwards-compatible).
- **FR-013**: When a user sets the override (FR-012) to a positive value that is smaller than what the auto-derived ceiling would have produced for the slot's current configuration, the plugin MUST accept the user's value without warning (this is the user's explicit "I want a smaller buffer than auto-derivation suggests" choice, accepting that US1's duration contract may not be met). When the user sets the override to a value that the host cannot allocate at slot start time, FR-006's clamp-and-warn path applies symmetrically (the same clamp, warning, and actionable-next-step wording as the auto-derived case). The override MUST NOT silently change the user-configured `replay_seconds` value — the time bound from FR-009 stays as-is regardless of override.
- **FR-014**: The plugin MAY sample observed encoded bitrate from the replay-buffer output during slot runtime (for example, by computing deltas of `obs_output_get_total_bytes` over a window — the precise mechanism is plan-phase). When such a measurement is available at the moment a Save Replay log line is emitted, the plugin SHOULD include it in the save log so the user can compare the auto-derived ceiling's bitrate assumption against the actually observed bitrate. The plugin MUST NOT use the observed bitrate to resize the buffer mid-session; the ceiling remains deterministic from configuration per FR-004. The observed bitrate is informational only and exists to help the user decide whether to set the FR-012 override on the next slot start.

### Key Entities *(include if feature involves data)*

- **Slot configuration (replay-buffer-relevant fields)**: The persisted set of fields for one recording slot, restricted here to those that influence replay-buffer memory sizing. These are: video resolution (width × height), framerate (fps_num / fps_den), rate-control mode string, rate-control numeric value, audio bitrate, audio track count, configured replay duration (seconds), and container format. For consumer slots, the rate-control mode and value used for sizing are the owner's effective values per feature 006's FR-014, not the consumer's own stored fields.
- **Effective rate-control state**: Already defined in feature 006. For this spec, the effective mode determines which sizing branch is taken (bitrate-based: existing formula; quality-based: new formula introduced by this spec). The effective value is consumed by the bitrate-based branch as `kbps`; the quality-based branch does NOT consume it (since a quality value is not a bitrate).
- **Replay buffer memory ceiling**: The `max_size_mb` value passed to the OBS replay-buffer output at slot start. Today derived solely from `est_kbps × replay_seconds × safety_factor` with a 50 MB floor. After this fix, derived from a configuration-dependent formula whose quality-based branch produces a ceiling generous enough to hold the configured duration under typical worst-case scene complexity for the configured resolution and fps — unless the per-slot override field (Key Entity below) is set, in which case the user-supplied value is used verbatim.
- **Per-slot replay buffer size override**: A new persisted per-slot configuration field, holding an optional integer in megabytes. When unset / empty / 0, the plugin's auto-derived ceiling is used; when set to a positive integer, the user-supplied value is used as `max_size_mb` instead. Defaults to unset for slots created before this feature shipped (backwards-compatible). Editable in the slot editor with the currently-resolved ceiling shown alongside.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For 100% of Save Replay operations triggered against a slot configured for a quality-based rate-control mode and a replay duration of N seconds, the saved file's duration is within one keyframe / GOP interval of N — i.e., the configured duration is honored, and the user's clip ends approximately where the user expected it to end. Exceptions are explicitly observable rather than silent: the **FR-006 clamp path** (host RAM insufficient at slot start) surfaces via the L2 warning, and the **FR-011 runtime cap-bound case** (encoded bitrate exceeds the auto-derived ceiling's assumption during the buffered window) surfaces via the L4 hedged warning at save time. The SC-001 guarantee is therefore "honored unless the user is told otherwise" rather than unconditional.
- **SC-002**: Across multiple consecutive Save Replay triggers against the same slot in the same session under content of differing scene complexity, the saved file durations are approximately equal (each within one keyframe / GOP interval of the configured duration). No "this clip is short because there was more motion" outcome occurs.
- **SC-003**: The continuous recording output's behavior is unchanged by this work, verified by inspection: the recording path is not in this code's scope and is not modified.
- **SC-004**: For every slot start that enables the replay buffer, the user can determine the per-slot memory commitment from the plugin log alone, without consulting source code or running profiling tools. Sum across all running slots equals approximately the total memory the host commits to replay buffers.
- **SC-005**: Per-slot memory commitment for the replay buffer is a deterministic function of the slot's configuration — two slots with identical configurations produce identical commitments, and a user can predict the commitment from the configuration before starting the slot. This guarantee applies **absent the FR-006 clamp path**: when host-reported available physical RAM is insufficient and the pre-emptive sizing check clamps the requested ceiling, the resolved commitment varies with host RAM state and is no longer predictable from configuration alone — but in that case the FR-006 clamp-and-warn log line makes the deviation observable, so the user can attribute the difference correctly rather than seeing it as silent non-determinism.
- **SC-006**: When the configured per-slot ceiling cannot be allocated at slot start, the resulting deviation is local — only the affected slot is impacted, and the slot's replay buffer is either clamped to a smaller ceiling (FR-006 clamp-and-warn branch) or declined entirely (FR-006 decline branch when the post-clamp ceiling falls below the 50 MB defensive floor). In both cases the deviation is observable in the log with enough detail for the user to act (slot name, requested ceiling, clamped or declined value, reason, actionable remedies), and no other slot's startup is affected.
- **SC-007**: Bitrate-based rate-control modes (CBR, VBR, ABR, etc.) continue to honor configured replay duration with the same correctness they exhibit before this fix — verified by regression testing the existing bitrate-mode path against the same Save Replay scenarios used for the quality-mode path.

## Assumptions

- The bug is specific to the per-slot replay buffer's `max_size_mb` derivation at `src/slot.cpp:891` and `src/slot.cpp:905`. "Replay" in this spec refers only to the per-slot replay buffer; it does not refer to OBS Studio's separate main replay buffer feature.
- The set of rate-control modes classified as "quality-based" for sizing-branch selection is exactly the inverse of `rc_util::is_bitrate_based(mode)` at `src/slot.cpp:22-30` — i.e., everything that returns false from `is_bitrate_based`, which today is CQP, CRF, CQ, ICQ, QVBR, and Lossless. If a new rate-control mode is introduced to `is_bitrate_based` later, this fix's branch selection automatically follows; no separate enumeration is duplicated.
- The OBS replay-buffer output (`replay_buffer`) accepts a `max_size_mb` setting in megabytes (long-long), enforces it as an upper bound on the in-memory ring, and evicts older frames when the bound is reached. The configured `max_time_sec` is a separate upper bound; the buffer evicts when either bound is hit. This spec's quality-mode sizing branch produces an `max_size_mb` generous enough that under typical conditions `max_time_sec` is the binding constraint.
- The "configured duration is honored within one keyframe / GOP interval" tolerance in FR-002 and SC-001 matches the existing behavior of bitrate-mode replay saves today: the OBS replay buffer evicts on a keyframe boundary, so the saved file's exact duration is within one GOP of the configured `max_time_sec`. This spec does not tighten that tolerance; it only requires the quality-mode case to meet the same tolerance as the bitrate-mode case. **Typical GOP interval**: for the maintainer's reference scenario (NVENC P5 at 1080p60) and for the common hardware-encoder defaults across NVENC / QSV / AMF / VideoToolbox / `obs_x264` at OBS's shipped defaults, the keyframe interval is `≤ 2 seconds` (most encoders default to 250 frames at 60 fps ≈ 4.17 s, but OBS-side recording-output defaults override this to 2 s; some hardware encoder presets emit IDRs more frequently still). For verification, a saved replay of `N` seconds is "within tolerance" if the file's duration is in the range `[N − 2, N + 2]` seconds — assuming the GOP interval has not been manually raised by the user. If a user has explicitly configured a larger keyframe interval (e.g., `keyint_sec = 5` for streaming-optimized encoder settings), the tolerance widens correspondingly; this is the user's choice and not a regression of this spec. Quickstart.md's T1 / T2 verification thresholds use the `± 2 s` window as the pass/fail criterion.
- The plan-phase decision for the quality-mode sizing formula is constrained by FR-001 / FR-001a / FR-001b (resolved from clarification Q3): per-mode branches (separate bitrate / quality / lossless paths), 2× margin, quality-mode estimate scales with resolution × fps (per-pixel-per-frame heuristic, not a flat Mbps constant), lossless gets its own higher estimate. The spec does NOT bind specific coefficients; the plan picks them and validates against the maintainer's reference scenario — CQP-17 at 1080p60 on NVENC P5 / full-res double-pass / high-quality tuning / high profile, running a modern open-world third-person shooter, where the encoder has been observed peaking at 60+ Mbps. A 40 s buffer of that content needs approximately 60 × 40 × 2 / 8 ≈ 600 MB to actually hold 40 s on disk; the plan-chosen coefficients MUST produce at least that ceiling for that scenario. Configurations the auto-derived default still under-shoots are addressed by the FR-012 user override, matching the OBS-native replay-buffer "default + override" model.
- The fix is verified manually on Windows (the maintainer's test environment) using the encoders available there (typically NVENC + `obs_x264`; AMF / QSV / VideoToolbox where the hardware exists). macOS and Ubuntu builds receive the same code path; cross-platform correctness rests on the platform-agnostic nature of the buffer-sizing computation (it is an integer arithmetic operation passed to libobs's replay-buffer output settings, with no platform-specific branches) and on future community reports.
- Feature 007's truthful-save logging (007's FR-011 / FR-012 / FR-013) is in place when this fix ships. **Identifier disambiguation**: feature 007 and this spec both use the identifiers FR-011, FR-012, and FR-013, but they mean different things. 007's FR-011 / FR-012 / FR-013 govern save-success-vs-failure wording in the replay-save log line. This spec's FR-011 / FR-012 / FR-013 govern, respectively: the cap-bound truncation hedged warning at save time, the per-slot "Max replay buffer size (MB)" override in the editor, and the override's accept-without-warn semantics. Throughout this document, an unprefixed `FR-NNN` refers to this spec's requirement; references to 007's identifiers are always prefixed with `007's` or `feature 007's`. This spec's FR-011 ("identify memory-cap eviction as the cause") complements 007's existing wording rules — it adds a *post-success diagnostic* for the case where the save produced a file but the file is shorter than configured. This is distinct from 007's *failure-vs-success* wording rules and does not alter them.
- Feature 006's FR-014 (consumer-side reads of rate-control resolve to the owner's effective values) is in place when this fix ships. The existing `eff.mode` / `eff.value` parameters at `src/slot.cpp:891` already carry the owner's effective values for consumer slots; this fix consumes them as-is and does not introduce a new owner-resolution path.

## Out of Scope

- Continuous recording output behavior: not modified.
- Replay file naming and uniqueness: handled by feature 007; not modified here.
- Editor UI: the user does not gain new controls for replay duration or rate-control mode beyond what already exists. One new control IS added — a per-slot writable "Max replay buffer size (MB)" override per FR-012 (resolved from clarification Q1 in this session). The fix is otherwise under-the-hood; the new diagnostic log lines (FR-005, FR-011) and the new editor control (FR-012) are the user-visible surface added.
- OBS Studio's main replay buffer (the one configured in OBS Settings → Output → Replay Buffer): not in this plugin's scope.
- Changes to which rate-control modes are classified as bitrate-based vs quality-based: out of scope; this spec uses `rc_util::is_bitrate_based` as-is.
- Adapting the buffer ceiling at runtime based on observed bitrate: out of scope per FR-004 and clarification Q2 (the ceiling MUST be deterministic from configuration; runtime growth would require stopping the replay output and discarding the buffered window, defeating the point). Measuring observed bitrate for diagnostic / logging purposes IS in scope per FR-014 — the plugin samples the rate but never resizes. A future spec MAY introduce a runtime-adaptive path if FR-004's deterministic ceiling combined with the FR-012 user override proves insufficient in practice.
